# PatentForge v0.4 — AI-Assisted Claim Drafting

## Overview

v0.4 adds a new Python + LangGraph service that generates patent claim drafts from the upstream feasibility analysis and prior art research. Three AI agents (Planner, Writer, Examiner) collaborate to produce independent claims at three scope levels, dependent claims, and supporting specification language. All output is framed as research — not legal advice — with stronger UPL guardrails than the feasibility pipeline.

## What v0.4 Delivers

1. **Claim drafting service** — Python + LangGraph multi-agent pipeline
2. **Three independent claims** — broad, medium, and narrow scope, informed by prior art and §102/§103 findings
3. **Dependent claims** — hierarchical claims derived from each independent claim
4. **Supporting specification language** — generated description text that supports the claims
5. **User-editable claim text** — edit any claim before moving to compliance review
6. **Stronger UPL guardrails** — explicit acknowledgment modal, DRAFT watermarks, claim-level disclaimers

## What v0.4 Does NOT Deliver (Deferred)

- Claim tree visualization → v0.4.1
- Compliance checking (§112 definiteness, §101 eligibility) → v0.5
- Individual claim regeneration → v0.5
- Prior art overlap warnings inline with claims → v0.5
- Patent family tree lookup → v0.4.1

### Three-Agent Pipeline

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ PLANNER  │────▶│  WRITER  │────▶│ EXAMINER │
│          │     │          │     │          │
│ Analyzes │     │ Drafts   │     │ Reviews  │
│ prior art│     │ claim    │     │ for weak │
│ + §102/  │     │ text at  │     │ points,  │
│ §103 to  │     │ 3 scope  │     │ §112     │
│ plan     │     │ levels + │     │ issues,  │
│ claim    │     │ deps     │     │ prior art│
│ strategy │     │          │     │ overlap  │
└──────────┘     └──────────┘     └──────────┘
                                       │
                                       ▼
                                  ┌──────────┐
                                  │  WRITER  │ (revision pass)
                                  │ Revises  │
                                  │ based on │
                                  │ examiner │
                                  │ feedback │
                                  └──────────┘
```

**LangGraph state machine flow:**
1. `plan` → Planner analyzes inputs, produces claim strategy (scope boundaries, claim types, key limitations)
2. `draft` → Writer generates initial claims (3 independent + dependents) following the plan
3. `examine` → Examiner critiques the draft against prior art, flags §112 issues, scores each claim
4. `revise` → Writer revises claims based on examiner feedback
5. `finalize` → Output structured claim set with metadata

The Examiner can send claims back for one revision cycle. After revision, the pipeline finalizes regardless. This prevents infinite loops while still getting one round of self-critique.

### Input: What the Claim Drafter Receives

From the backend (via the adapter), the claim drafter receives:

```python
class ClaimDraftRequest(BaseModel):
    invention_narrative: str           # Original invention form → narrative
    feasibility_stage_5: str           # IP Strategy & Recommendations
    feasibility_stage_6: str           # Consolidated Report
    prior_art_results: list[PriorArt]  # Top-scored prior art with titles/abstracts
    prior_art_claims: dict[str, str]   # Patent number → claims text (from ODP)
    settings: DraftSettings            # API key, model, max tokens
```

### Output: What the Claim Drafter Produces

```python
class ClaimDraftResult(BaseModel):
    claims: list[Claim]
    specification_language: str     # Supporting description text
    planner_strategy: str           # Planner's reasoning (visible to user)
    examiner_feedback: str          # Examiner's critique (visible to user)
    revision_notes: str             # What changed in revision

class Claim(BaseModel):
    claim_number: int
    claim_type: Literal["INDEPENDENT", "DEPENDENT"]
    scope_level: Literal["BROAD", "MEDIUM", "NARROW"] | None  # Independent only
    parent_claim_number: int | None  # Dependent only
    text: str
    examiner_notes: str             # Per-claim examiner commentary
```

### Backend Integration

New adapter: `backend/src/adapters/claim-draft.adapter.ts`

- Receives "draft claims" request from controller
- Collects inputs: invention narrative, stage 5+6 outputs, prior art with claims
- POST to claim drafter service at `CLAIM_DRAFTER_URL` (default http://localhost:3002)
- SSE stream for progress events (plan_complete, draft_complete, examine_complete, revise_complete, finalize_complete)
- Saves results to existing `ClaimDraft` and `Claim` Prisma models

New API endpoints (already spec'd in ARCHITECTURE.md):
```
POST   /projects/:id/claims/draft       Trigger claim generation
GET    /projects/:id/claims              Get latest claim draft
GET    /projects/:id/claims/:version     Get specific version
PUT    /projects/:id/claims/:version     Update claims (user edits)
```

## UPL Guardrails (Stronger than Feasibility)

1. **Per-project acknowledgment modal** before first claim generation (not just first-run clickwrap)
2. **"DRAFT — NOT FOR FILING" watermark** on every claim in UI, HTML export, and DOCX export
3. **Examiner agent** explicitly trained to flag overreach and recommend attorney review
4. **Planner prompt** frames output as "claim research directions" not "patent claims"
5. **Per-claim disclaimer** in the examiner notes: "This draft claim has not been reviewed by a patent attorney"
6. **Export watermark** stronger than feasibility: "DRAFT CLAIM CONCEPTS — NOT REVIEWED BY AN ATTORNEY — NOT FOR FILING"
7. **All prompt files** licensed CC BY-SA 4.0 (disclaimers survive forks)

## Decisions (Resolved)

### 1. Model selection for agents
**Decision:** Planner uses `researchModel` (cheaper, analytical work). Writer and Examiner both use `defaultModel`. No separate examiner model setting — the examiner's quality comes from the prompt, not a bigger model. Users who want higher quality set their default to Opus.

### 2. Claim count
**Decision:** 20 total claim cap (matches USPTO fee boundary). Planner determines distribution based on invention complexity. Simple inventions: ~12-15 claims. Complex: ~18-21. The Writer prompt enforces the cap. This mirrors real filing practice.

### 3. Statutory claim types
**Decision:** Each of the 3 independent claims uses a different statutory type. Planner picks the best mapping based on invention type:
- Software/AI → method (broad) + system (medium) + CRM (narrow)
- Hardware/IoT → method (broad) + system (medium) + apparatus (narrow)
- Process → method (broad) + apparatus (medium) + composition (narrow)

One independent per type. Dependents inherit parent's type. Stays within the 3-independent fee boundary.
